into ONE Person entity. A person is a registrant until their DAY-1 FIRST
CHECK-IN, at which point they are promoted to a camper (registered -> arrived).
Staged into verifiable, additive steps because of the no-compiler constraint;
the destructive switchover (Step 4) is intentionally deferred.

PHASE 4: SECURITY HARDENING  (2026-06-18, in progress)
-------------------------------------------------------------------------------
- FIXED the #1 deployment blocker: replaced the in-memory token Map with STATELESS
  HMAC-SIGNED SESSIONS (auth.service). The old Map was fatal on serverless/multi-
  instance — each cold start began empty (logging everyone out) and a token minted
  on one instance was unknown to another. Tokens are now "<base64url payload>.<sig>"
  carrying the full actor (no DB lookup per request); SESSION_SECRET env var drives
  the HMAC with a prod startup warning if unset. logout() is now a client-side
  no-op (stateless tokens can't be server-revoked — the documented trade-off; they
  expire after the 12h TTL). auth.service.test updated for the new token shape +
  added tamper-rejection and expiry cases.
- Added HTTP SECURITY HEADERS (X-Content-Type-Options, X-Frame-Options DENY,
  Referrer-Policy, Permissions-Policy) and a CORS '*'-in-production startup warning
  (express-adapter). Set trust proxy so req.ip honours X-Forwarded-For.
- Added LOGIN RATE LIMITING: 10 attempts / IP / 15-min window -> 429 + Retry-After
  (ported RateLimiter util from CMS). In-memory, so per-instance (acceptable brute-
  force backstop; a shared store would be needed for a hard global limit — see R5).
  Harness: check_imports ✓ (44 files), sim_phase4 ✓ (11/11, HMAC validated vs
  Python hmac.compare_digest), no tokenStore refs remain, no unused imports.
  NOTE: kept scrypt password hashing (cryptographically sound; no existing hashes
  to migrate, so bcrypt+rehash from CMS was not needed under the security-parity
  decision). RLS + error sanitization still pending (RLS needs the Supabase layer).
